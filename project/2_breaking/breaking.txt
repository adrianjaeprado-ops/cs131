=== Phase 2 - Excel Breaking Point ===

Test file: combined_4days.csv (4 days of Q3 2025 data: July 1-4)
Rows: 1,289,227
Size: 483.2 MB

Excel's row limit is 1,048,576 rows. Our combined file exceeds this by 240,651 rows.

Result: Excel displayed an error on open: "This data set is too large for the Excel grid. If you save this workbook, you'll lose data that wasn't loaded." (see excel_failure.png)

This confirms Excel cannot load the full file once it exceeds the row limit, and any rows past the cutoff are simply left out. This means any analysis run in Excel on this dataset would be working with a partial and possibly unrepresentative slice of the data. This is exactly the kind of failure mode that motivates using command line tools and Spark instead.

=== Phase 2 - Pandas Breaking Point ===

Test file: combined_4days.csv (4 days of Q3 2025 data: July 1-4)
Rows: 1,289,226
File size on disk: 483.2 MB

Result: pandas.read_csv() successfully loaded the file, but took 67.06 seconds and consumed 9,601.41 MB (about 9.6 GB) of memory once loaded, roughly 20 times the file's size on disk.

This demonstrates the core problem with tools like pandas that load everything into memory, even when they do not outright crash: pandas expands each row into Python objects with significant overhead per cell, so memory usage grows far faster than file size. If we apply this same ratio to our full 3 quarter dataset (about 33.5 GB on disk across Q3 2025, Q4 2025, and Q1 2026), pandas would likely need somewhere in the range of 600 to 700 GB of RAM to load the entire dataset at once, which is far beyond what a normal laptop or even most single machines can provide. This is the direct motivation for using PySpark, which processes data across multiple machines rather than loading everything into memory on one machine at once.
